# Spec: Security Hardening S3–S6

**Data:** 2026-05-05
**Autor:** Aegis security review
**Status:** Draft → para implementação imediata
**Escopo:** S3 (encryption-at-rest), S4 (claim challenge-response), S5 (rate limit distribuído), S6 (replay protection por endpoint)

---

## 0. TL;DR

Quatro frentes de hardening, todas parcialmente endereçadas em sprints anteriores mas com lacunas conhecidas. O objetivo deste spec é fechar todas as lacunas em **um único pull-request**, com testes unitários e de integração cobrindo cada caminho, sem quebrar nenhum fluxo existente (incluindo invoices legadas pré-encryption e pré-`signPubkey`).

| Item | Estado atual | Lacuna principal |
|---|---|---|
| **S3** Cifrar UTXO secrets | `field-crypto.ts` existe; `utxoPrivateKey` e `utxoBlinding` cifrados em PATCH | Linhas legadas em texto plano não migradas; `GET /api/stealth/[id]` decifra `utxoBlinding` para todos os membros do vault |
| **S4** Challenge-response no claim | `claim-challenge.ts` (HMAC stateless) + `signPubkey` em invoices novas | Legacy invoices sem `signPubkey` **pulam verificação de assinatura**; `consumeChallenge` é no-op → replay possível dentro do TTL de 60s; endpoint `/challenge` sem rate-limit |
| **S5** Rate limit distribuído | `checkRateLimitAsync` com Redis (Upstash) | Race entre `INCR` e `EXPIRE`; bucket somente por IP; limites hardcoded `10/min` em todos os endpoints; sem atomicidade Lua |
| **S6** Replay protection por endpoint | Não implementado | `X-Solana-Signature` autentica apenas `aegis:{pubkey}:{ts}:{nonce}`; sig captada num endpoint vale em qualquer outro durante 5 min |

## 1. Contexto

### 1.1 O que já foi feito (sprints S1–S2)

- **S1:** wallet auth via signed message (`aegis:{pubkey}:{ts}:{nonce}`), nonce somente para uniqueness do payload (sem rastreio server-side); cofre operator gating em endpoints sensíveis.
- **S2:** `requireVaultMember` baseado em on-chain state via slice byte-offset corrigido (40,72).
- **Field encryption:** AES-256-GCM com chave derivada de `JWT_SIGNING_SECRET` via SHA-256, IV aleatório de 12 bytes, authTag de 16 bytes; format `base64(iv || ciphertext || authTag)`.
- **Stateless challenge:** HMAC-SHA256 sobre `invoiceId|nonce|expiresAt` no token; cliente assina nonce com Ed25519 derivado de `seed = boxKp.secretKey.slice(0,32)`.

### 1.2 Por que não está terminado

A entrega de S1/S2 priorizou gates de autorização (problemas P0 de exposição de dados). As proteções secundárias (encryption-at-rest completa, replay binding, rate limit per-route) ficaram com implementações parciais que precisam fechar antes do mainnet.

## 2. S3 — Cifrar UTXO secrets no banco

### 2.1 Estado atual

- `apps/web/lib/field-crypto.ts` exporta `encryptField`, `decryptField`, `isEncrypted`.
- `PATCH /api/stealth/[id]/utxo` cifra `utxoPrivateKey` e `utxoBlinding` antes de gravar.
- `POST /api/stealth/[id]/claim-data` decifra ambos quando `isEncrypted()` retorna `true`.
- `GET /api/stealth/[id]` decifra `utxoBlinding` (mas **não** `utxoPrivateKey`).
- `isEncrypted()` é heurística: retorna true se há char `[A-Z+/]` (hex puro nunca tem; base64-GCM sempre tem).

### 2.2 Problemas

**P1 — Sobre-exposição de blinding:** `GET /api/stealth/[id]` retorna `utxoBlinding` decifrado para qualquer membro do vault. `utxoBlinding` faz parte da preimage do commitment Cloak; expor para todos os signatários reduz a privacidade do destinatário. Apenas o operador (responsável por gerar os UTXOs) e o destinatário (no claim-data) precisam dele.

**P2 — Linhas legadas:** Invoices criadas antes do PATCH com encryption têm `utxoPrivateKey` em hex puro. Embora `isEncrypted()` trate o caso, a coluna não está uniforme e não há migration script para cifrar retroativamente. Em uma rotação de chave (ver P3), esses dados ficariam órfãos.

**P3 — Rotação de chave:** A chave deriva de `JWT_SIGNING_SECRET`. Se o secret rotacionar, todo dado existente fica ilegível. Não há suporte a multi-key (key id no envelope).

**P4 — `isEncrypted()` heurística:** Funciona hoje porque hex é o único formato legacy. Se algum legacy for base64 (improvável mas possível), causa falso positivo. Bug 7 do sprint 1 já mostrou isso. Solução robusta = prefixar envelope com versão (`v1:base64...`).

### 2.3 Mudanças

**S3.1 — Envelope versionado.**
Trocar formato de saída de `encryptField` para `v1.{base64(iv||ct||tag)}`. `decryptField` aceita ambos os formatos (com e sem prefixo) durante a transição. `isEncrypted()` passa a checar prefixo `v1.` (mais robusto que regex de char).

**S3.2 — Não expor `utxoBlinding` em `GET /api/stealth/[id]`.**
Remover `utxoBlinding` da resposta do listing. Membros do vault que precisam reconstruir o commitment para auditoria devem ler do `utxoCommitment` (que já é público on-chain). O blinding fica acessível apenas a:
- Operador (PATCH /utxo — ele gerou)
- Destinatário (POST /claim-data — após challenge-response válido)

**S3.3 — Migration de back-fill.**
Adicionar migration `20260505100000_encrypt_legacy_utxo`:
- Lê todas as linhas com `utxoPrivateKey IS NOT NULL` ou `utxoBlinding IS NOT NULL`.
- Para cada linha, se valor não começa com `v1.`, cifra e atualiza.
- Roda como script Node em `apps/web/prisma/scripts/encrypt-legacy-utxo.ts` que o time roda manualmente após `prisma migrate deploy`. Justificativa: SQL puro não consegue rodar AES, e Prisma migration de dados é frágil.

**S3.4 — Documentar rotação de chave.**
Adicionar nota em `field-crypto.ts` sobre a estratégia de key rotation (futura): trocar prefixo para `v2.{kid}.{base64...}` quando precisar; manter `JWT_SIGNING_SECRET_PREVIOUS` em env durante transição.

### 2.4 Testes

- `tests/unit/field-crypto.test.ts` (novo):
  - `encrypt → decrypt` round-trip.
  - `decryptField` aceita formato legacy (sem prefixo) e v1.
  - `isEncrypted` retorna `false` para `null`, hex puro, string vazia; `true` para `v1.…`.
  - Tampering: alterar 1 byte do ciphertext faz `decryptField` lançar (authTag).
  - Chave errada: `decrypt` com `JWT_SIGNING_SECRET` diferente lança.

- `tests/integration/stealth-encryption.test.ts` (novo):
  - PATCH /utxo grava cifrado (verifica raw row no DB tem prefixo `v1.`).
  - GET /api/stealth/[id] não retorna `utxoBlinding`.
  - POST /claim-data decifra corretamente.

## 3. S4 — Challenge-response no claim

### 3.1 Estado atual

- `POST /api/stealth/[id]/challenge` retorna `{ challengeId, challenge }` (HMAC + nonce 32B).
- `POST /api/stealth/[id]/claim-data` exige `challengeId + derivedPubkey + challengeSignature`, valida HMAC + Ed25519 sobre nonce.
- `signPubkey` (pubkey de assinatura derivada do mesmo seed do box) gravada na invoice no momento de criação.
- Migration `20260503000000_stealth_sign_pubkey` adicionou coluna.

### 3.2 Problemas

**P1 — Bypass para invoices legadas:** se `invoice.signPubkey` é `null`, `claim-data` **pula a verificação de assinatura** (linhas 83–86 do claim-data/route.ts). Significa que, para qualquer invoice antiga, basta ter `recipientWallet auth` + `derivedPubkey == invoice.stealthPubkey` para baixar os dados. Isso vaza o blinding/private key se um atacante conseguir um wallet auth do destinatário e ler o `stealthPubkey` (que é público).

**P2 — `consumeChallenge` é no-op:** o token é stateless (HMAC), então a "consumação" comentada como API compat na verdade não invalida nada. Um challenge pode ser usado múltiplas vezes durante o TTL de 60s. Risco prático: baixo (precisa do signing key derivado), mas a property "one-time use" prometida no comentário não é honrada.

**P3 — `/challenge` sem rate-limit:** atacante pode emitir milhões de challenges para qualquer invoice (cria carga no DB e atrita). Sem auth, sem limite.

**P4 — Fragmento `#sk=` em URL de claim:** `POST /api/stealth` retorna `claimUrl = /claim/${id}#v=1&sk=${secret}&vault=${cofre}`. O fragmento (`#`) **não é enviado em request HTTP** ao servidor, então não loga em access logs. Esse vetor está mitigado por design — não é um bug, mas vale documentar.

### 3.3 Mudanças

**S4.1 — Forçar `signPubkey` em todas as invoices.**
- Migration `20260505110000_backfill_sign_pubkey`: para invoices legadas sem `signPubkey`, marcar `status = 'voided'` e `memo = 'pre-S4 voided for security upgrade'`. Justificativa: não dá para derivar `signPubkey` retroativamente sem o seed (que está no fragment do claim link, não no banco). Voiding força o emissor a re-emitir a invoice. Aceito porque o número de invoices legadas em devnet é pequeno.
- Em `claim-data/route.ts`, remover o branch `if (!invoice.signPubkey)` — exigir sempre verificação de signature.

**S4.2 — One-time challenge real (Redis-backed).**
- Adicionar `consumeChallenge(invoiceId, challengeId)` que faz `SET nonce-used:{invoiceId}:{sha256(challengeId)} 1 EX 120 NX`. Se retorna `null` (já consumido), retorna falha.
- Em devnet/local sem Redis, manter no-op (degradação aceita; risco baixo). Variar via `process.env.REDIS_URL`.
- TTL=120s (2× challenge TTL) para garantir que entra a janela inteira de validade.

**S4.3 — Rate-limit `/challenge`.**
- Bucket por `invoiceId` (não IP) com limite 10 challenges/min. Justificativa: limit by-IP penaliza usuários atrás de NAT corporativo; limit by-invoice limita o ataque ao alvo real.
- Reusar `checkRateLimitAsync` com chave `chal:{invoiceId}`.

### 3.4 Testes

- `tests/integration/claim-challenge.test.ts` (novo):
  - Fluxo feliz: emite challenge → assina nonce → claim-data retorna UTXO.
  - Reutilizar challenge → 403 (S4.2).
  - Challenge expirado (mock Date.now) → 403.
  - Challenge ID adulterado (HMAC quebra) → 403.
  - `signPubkey` ausente → 403 (regressão pós-S4.1; legacy invoices voided).
  - Rate-limit `/challenge`: 11ª request em 60s para mesmo invoiceId → 429.

## 4. S5 — Rate limiting distribuído

### 4.1 Estado atual

- `checkRateLimitAsync(ip, limit, windowMs)`: `INCR` + `EXPIRE` no Redis se `REDIS_URL`, fallback in-memory caso contrário.
- Todas as 7 routes que usam: limite default 10/min por IP.
- `@upstash/redis` carregado dinamicamente; fallback para REST API por fetch.

### 4.2 Problemas

**P1 — Race entre `INCR` e `EXPIRE`.** Se o processo cair entre os dois comandos, a chave fica sem TTL → leak de memória no Redis (eternamente preserva contadores). Para Upstash REST esse risco é menor (cada call HTTP é isolada), mas existe.

**P2 — Bucket somente por IP.** Endpoints autenticados deveriam bucket por wallet pubkey também. Hoje, atacante alternando IPs (proxies/VPNs) bypassa o limite. Pior: atrás de NAT corporativo, todos os usuários compartilham o IP — um usuário rate-limita os colegas.

**P3 — Limites hardcoded.** Todas as routes 10/min. `/api/stealth/[id]/challenge` deveria ser mais permissivo (cliente assina rapidamente em sequência), `/api/proposals` mais restritivo (escrita on-chain).

**P4 — Sem fixed-window robusto.** Algoritmo é fixed-window simples: a janela "reseta" no boundary, permitindo até 2× burst em 1 segundo (10 reqs no segundo 59 + 10 no segundo 61). Aceitável para os limites em vigor; documentar.

### 4.3 Mudanças

**S5.1 — Atomicidade via single-call SET NX EX + INCR.**
Substituir o par `INCR` + `EXPIRE` por:
```ts
// Pseudo-Lua, traduzido para 2 calls com NX:
const setOk = await redis.set(key, "1", { ex: windowSecs, nx: true });
if (setOk === "OK") return true; // primeiro hit, contador = 1
const current = await redis.incr(key);
return current <= limit;
```
`SET NX EX` garante que TTL é setado **junto** com o valor inicial. Compatível com REST e SDK. Cobre P1.

**S5.2 — Bucket composto.**
Refatorar para `checkRateLimitAsync(bucket: string, limit, windowMs)` onde o caller monta o bucket. Helpers:
- `rateLimitKeyFor(req, scope)` → `${ip}:${pubkey ?? 'anon'}:${scope}`.
- Routes autenticadas usam pubkey; públicas (challenge, vaults/[multisig] read) usam IP.

**S5.3 — Limites por route.**
Tabela em `apps/web/lib/rate-limit.ts` com perfis nomeados:
```ts
export const RATE_LIMITS = {
  default: { limit: 30, windowMs: 60_000 },     // bumped from 10
  write: { limit: 10, windowMs: 60_000 },       // POST/PATCH/DELETE
  challenge: { limit: 20, windowMs: 60_000 },   // /challenge (per invoiceId)
  signature: { limit: 60, windowMs: 60_000 },   // claim-data, signed reads
} as const;
```
Cada route declara qual perfil usa.

**S5.4 — Documentar fallback in-memory.**
Adicionar `console.warn` (uma vez por boot) quando `REDIS_URL` não está setado em produção (`NODE_ENV === 'production'`). Em dev, silencioso.

### 4.4 Testes

- `tests/unit/rate-limit.test.ts` (novo):
  - In-memory: 10 hits passam, 11º falha; após `windowMs`, reseta.
  - In-memory: buckets diferentes não interferem.
  - Redis (mock): SET NX EX é chamado uma vez no primeiro hit; INCR nas seguintes.
  - Redis (mock): após TTL, novo SET NX EX (não acumula).

- Não testar Upstash real em CI (latência + flakiness). Mock via `vi.mock("@upstash/redis", …)`.

## 5. S6 — Replay protection por endpoint

### 5.1 Estado atual

Cliente assina `aegis:${pubkey}:${timestamp}:${nonce}`. Servidor valida em `verifyWalletAuth` apenas:
- Pubkey válida
- Timestamp dentro de 5 min
- Assinatura Ed25519 válida sobre o payload

**Não há binding com método HTTP, path ou body.** Uma assinatura captada em (ex.) `GET /api/vaults/mine` pode ser replayada em `POST /api/proposals` com qualquer body durante 5 min.

### 5.2 Mudanças

**S6.1 — Novo formato de payload assinado.**

Trocar `message = aegis:{pubkey}:{ts}:{nonce}` por:
```
aegis:v2:{pubkey}:{ts}:{nonce}:{method}:{path}:{bodyHash}
```

Onde:
- `method` = uppercased (GET/POST/PATCH/DELETE).
- `path` = pathname normalizado (`/api/proposals/multisig/index`, sem query string).
- `bodyHash` = base64url(sha256(rawBodyBytes)) — para GET/HEAD sem body, `bodyHash = "-"`.

**S6.2 — Cliente: hook `useWalletAuth` precisa do contexto da request.**

Hoje `getAuthHeaders()` é chamado sem contexto. Vamos:
1. Adicionar `signRequest(method, path, body)` retornando headers.
2. `fetchWithAuth(input, init)` chama `signRequest(init.method ?? "GET", new URL(input).pathname, init.body)` internamente.
3. **Cache continua válido**: o cache de auth de 4 minutos era apenas para evitar repetir o `signMessage`. Com S6, **cada request precisa de uma nova assinatura** (porque method/path/body mudam). Cache vai pra fora.

   Trade-off: cada request = 1 wallet popup? **Não.** A assinatura é determinística e o navegador a faz silenciosamente sem prompt em wallets que suportam (`signMessage` para Phantom é silent só se o site pediu autorização via `connect`). Solução: documentar o trade-off e aceitar que cada API call mostre o popup, OU manter um esquema de "session key": ao conectar, gerar um keypair Ed25519 efêmero no client, assinar uma vez com a wallet (`session-grant:{sessionPubkey}:{exp}`), salvar na localStorage cifrada, e usar a session key para assinar cada request. **Decisão: usar session keys.** Mais complexo mas é o único caminho UX-aceitável.

   **Update após reconsideração:** session keys são uma camada nova com risco próprio. Para este sprint, alternativa mais simples: **manter o popup-uma-vez-por-N-min** mas trocar payload para incluir `bodyHash` derivado de um `request-id` (UUID gerado no client, válido 4 min, "amarrado" à wallet). O servidor exige `X-Request-Id` no header; uma sig+request-id só pode ser usado UMA vez (consume via Redis). Method+path ficam binding.

   **Decisão final do spec:** **Adotar payload `aegis:v2:{pubkey}:{ts}:{nonce}:{method}:{path}:{bodyHash}` com sig por-request.** Re-prompt no popup é aceitável SE for silent (Phantom é). Validar empiricamente (item de teste manual). Se popup for visível, fallback para session key vira S7.

**S6.3 — Backward compat.**

Aceitar simultaneamente:
- Header `x-solana-signature` no formato v1 (se message não tem `:v2:`, valida como antes).
- Header `x-solana-signature` no formato v2 (com binding).

Em produção, ambos válidos por 7 dias após release; depois remover v1. Controlado por flag `ALLOW_LEGACY_AUTH=true` em env (default: true, vira false após release de prod).

**S6.4 — Path normalization.**

Para evitar bypass via `/api/proposals/` vs `/api/proposals` ou `%2F`-encoded paths:
- Servidor usa `new URL(req.url).pathname`.
- Trim trailing slash.
- Toupper method.

### 5.3 Testes

- `tests/unit/wallet-auth.test.ts` (novo):
  - v2 round-trip: cliente assina, servidor verifica.
  - v2: alterar 1 char do body → invalida sig.
  - v2: replay em método diferente (GET → POST) → invalida.
  - v2: replay em path diferente → invalida.
  - v1 com `ALLOW_LEGACY_AUTH=true` → aceita.
  - v1 com `ALLOW_LEGACY_AUTH=false` → rejeita.

- `tests/integration/wallet-auth-routes.test.ts`:
  - POST /api/proposals com sig v2 válida → 200.
  - POST /api/proposals com sig v2 do path errado → 401.

## 8. Ordem de implementação

A ordem foi escolhida para evitar quebrar o app entre commits:

1. **S5 (rate limit):** zero side-effects no client. Refator interno.
2. **S3 (encryption):** envelope versionado + remoção do `utxoBlinding` em GET. Migration de back-fill rodada manualmente.
3. **S4 (challenge):** void de invoices legadas, exigência de `signPubkey`, one-time consume Redis-backed.
4. **S6 (replay binding):** mais arriscado — cliente E servidor precisam casar. Implementar com `ALLOW_LEGACY_AUTH=true`, validar manualmente, depois flip.

## 10. Riscos e mitigações

| Risco | Severidade | Mitigação |
|---|---|---|
| S3 migration corrompe linhas em prod | Alta | Script é idempotente (só toca linhas sem `v1.`); rodar com dry-run primeiro |
| S4 voida invoices ativas | Média | Apenas devnet; HANDOFF documenta; comunicar antes do deploy |
| S6 quebra cliente porque popup é visível | Média | Validar em dev antes do flip; `ALLOW_LEGACY_AUTH=true` permite rollback rápido |
| Redis indisponível em prod (S5) | Baixa | Fallback in-memory já existe; warn loud em prod |
| `JWT_SIGNING_SECRET` rotaciona | Alta | Documentar no `.env.example`; envelope v1 abre caminho para multi-key futuro |
