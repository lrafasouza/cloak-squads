## 10. Methodology Lessons (for the next audit cycle)

1. **Cross-pass seams find the bugs single-pass audits miss.** Pass 5 found 6 findings (1 Sev 7) at boundaries that each individual pass had already verdicted clean. Future audits should include an explicit cross-pass seam phase **before** declaring done.
2. **Auth tier × response shape is its own audit axis.** P5-F-502 was a per-route authorization-tier mismatch — the route was auth'd at `vault_member` but its response shape was appropriate for `vault_operator`. This class is invisible to a per-handler scope and to a per-pass scope. Add to future audit prompts: *for every route, for every response shape, the auth tier must match the data tier*.
3. **`env.ts` ↔ deploy-platform YAML is its own surface.** Two synchronisation gaps shipped in this audit cycle (`AUDIT_EXPORT_SIGN_KEY` prefix without dashboard guidance; `REDIS_URL` validation without `render.yaml` declaration). The fix is a `NODE_ENV=production` smoke test against a populated env loaded *the way the production container loads it* — not via `pnpm typecheck` or `next build` (neither of which exercises the runtime env path).
4. **Test boot/env changes against a real standalone server, not just typecheck + unit tests.** Documented in memory `feedback_test_boot_changes_locally.md` after the 3-commit fix loop on 2026-05-08. Carries over here.
5. **Solo-dev branch protection should land on master merge, not be deferred.** Pass 3 P3-F-009 is the single highest-leverage governance lift; `gh api` recipe in `docs/security/governance.md` makes it a one-command change.
